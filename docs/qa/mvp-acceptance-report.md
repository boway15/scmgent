## 1. 验收结论

| 结论 | 说明 |
|------|------|
| **有条件通过** | P0 核心链路可用，P1 模块基本达标；存在 PRD 与实现差异及少量体验缺陷，不阻塞 MVP 演示与妙搭迁移 |

**总体评价**：系统已形成「主数据 → 库存/销量 → 安全库存/补货 → PMC 计划 → 跟单 → 合规/FOB/看板」的完整业务闭环。本地算法替代 Dify 为 MVP 既定方案。补货亮灯、多仓、角色菜单等迭代能力已落地。

---

## 2. 验收方法

1. **文档对照**：逐模块核对 PRD 用户故事与 API/页面清单  
2. **接口冒烟**：`GET /api/dashboard`、`/api/products/sku-overview`、`/api/inventory/overview`、`/api/me/menus` 等返回 200  
3. **代码走查**：路由、任务、权限中间件、亮灯逻辑  
4. **演示路径**：家具 Demo CSV 导入 → 库存总览 → 补货建议 → PMC → 看板

## 4. 缺陷与风险清单

| 编号 |  severity | 描述 | 状态 |
|------|----------|------|------|
| D-001 | 高 | 商品主数据页 `updateSkuLight` 未定义导致白屏 | **已修复** |
| D-002 | 中 | PRD 写 Dify 计算安全库存，MVP 为本地算法 | 文档已知差异 |
| D-003 | 低 | FOB 物流追踪为预留 | 后续 Phase |
| D-004 | 低 | 无独立 E2E 自动化测试 | 建议妙搭导入后冒烟 |
| D-FOB-01 | 高 | `fixed` 压夜费/指定柜号按体积分摊 | **已修复**（2026-06-09） |
| D-FOB-02 | 高 | 未平账可确认批次 | **已修复**（2026-06-09） |
| D-FOB-03 | 高 | `confirmed` 批次无只读保护 | **已修复**（2026-06-09） |
| D-FOB-04 | 高 | 账单独有柜号仍可核算导致永久差额 | **已修复**（2026-06-09） |
| D-FOB-05 | 高 | 异常确认可不填归属主体 | **已修复**（2026-06-09） |
| D-FOB-06 | 中 | 重导账单清空异常审核成果 | P1：非 confirmed 合并保留 |
| D-FOB-07 | 中 | 费用规则无管理 UI | P1 PRD |
| D-FOB-08 | 低 | 种子规则仅初始化一次，已部署库不自动补规则 | P1 规则迁移 |
| R-001 | 中 | Docker 改前端需 rebuild，用户易误以为 restart 即可 | 见 `z-docker-ops.md` |
